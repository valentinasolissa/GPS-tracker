# Supabase setup

The tracker app expects a single table, `locations`, in your Supabase Postgres database. Rows store a plaintext `device_id` for filtering and a base64-encoded protobuf blob in `raw_payload`.

## Create the table

In the Supabase SQL editor, run:

```sql
create table if not exists public.locations (
  id bigint generated by default as identity primary key,
  device_id text not null,
  raw_payload text not null,
  created_at timestamptz not null default now()
);

create index if not exists locations_device_id_idx
  on public.locations (device_id);

create index if not exists locations_created_at_idx
  on public.locations (created_at desc);
```

Adjust types or add columns only if you also update `tracker_node.py` and `verify_cloud_data.py`.

## Row shape (application contract)

| Column | Written by app | Purpose |
|--------|----------------|---------|
| `device_id` | Yes | Human-readable filter (e.g. `VALENTINA-PI-4B`) |
| `raw_payload` | Yes | Base64-encoded `DeviceStatus` protobuf bytes |
| `id` | Auto | Used by verifier: `order("id", desc=True).limit(1)` |
| `created_at` | Auto (optional) | Audit; not required by current Python code |

## API keys

1. Open your project → **Settings** → **API**.
2. Copy **Project URL** → `SUPABASE_URL` in `tracker_app/.env`.
3. Copy the **anon** `public` key → `SUPABASE_KEY`.

The Python client uses the anon key. For production devices, prefer **Row Level Security (RLS)** with policies that only allow inserts from authenticated devices, or use a dedicated service role on a trusted backend—not embedded in Pi images.

## Row Level Security

If RLS is enabled on `locations` without policies, inserts from `tracker_node.py` will fail.

Example policy for development (allows anonymous insert and read—**not for production**):

```sql
alter table public.locations enable row level security;

create policy "Allow anon insert"
  on public.locations
  for insert
  to anon
  with check (true);

create policy "Allow anon select"
  on public.locations
  for select
  to anon
  using (true);
```

Tighten policies before exposing the project publicly. Typical production patterns:

- Inserts only when `device_id` matches a registered device list
- No broad `select` for anon; reads via authenticated dashboard or service role

## Verify in the dashboard

After running `python tracker_node.py`:

1. **Table Editor** → `locations` → confirm new row.
2. `raw_payload` should be a long base64 string.
3. Run `python verify_cloud_data.py` on the Pi to decode the same row.

## Common errors

| Error | Fix |
|-------|-----|
| `relation "locations" does not exist` | Run the CREATE TABLE SQL above |
| `new row violates row-level security policy` | Add insert policy or disable RLS for dev |
| `Invalid API key` | Regenerate anon key; update `.env` |
| Empty result in verifier | Send at least one row with `tracker_node.py` first |
